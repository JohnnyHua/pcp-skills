# PROJECT.md 基线设计

## 目标

为 PCP 增加一份稳定的项目基线文件：`PROJECT.md`。

它不是 `Worklog`、不是 `Backlog`、也不是 `Handoff`。  
它的作用是作为项目的长期事实层，帮助 PCP 在 intake、版本判断和主线收敛时始终有一个稳定依据。

## 为什么现在做

当前 PCP 已经有：

- intake / intake_adopt
- task / queue / proposal / review
- blueprint / handoff / concern / backlog

但还缺一份稳定的项目基线。  
如果没有这个基线：

- 每次 intake 都会重新解释项目
- 项目边界会随着 agent 的说法慢慢漂移
- 版本路线图虽然存在，但还没有一个稳定的项目载体

## 第一版设计

第一版只使用 `PROJECT.md`，不引入 `PROJECT.json`。

原因：

- 人更容易读
- AI 对 Markdown 也能稳定理解
- 少一份结构同步负担
- 更符合当前 `v0 / v0.1` 阶段“先把骨架跑起来”的目标

## 建议结构

`PROJECT.md` 第一版建议包含这些章节：

### 1. 项目基线

- 这个项目是做什么的
- 当前核心目标是什么

### 2. 架构概况

- 当前主要模块
- 关键目录 / 关键文件

### 3. 版本路线图

- `v0`：目标一句话
- `v0.1`：目标一句话
- `v0.2`：目标一句话
- `v0.3`：目标一句话

### 4. 当前状态

- 当前处于哪个版本
- 当前主线在做什么
- 当前版本还差什么

### 5. 项目约束

- 正式任务必须审批
- Blueprint 只用于复杂任务
- intake 先对齐认知，不自动开工

### 6. 刷新规则

- `PROJECT.md` 是基线，不是流水账
- 普通执行流不应随手改它
- 如果 agent 认为它过时了，只能提议刷新
- 用户同意后再更新

## 使用方式

### intake 时

`pcp_intake` 后续应该优先读取 `PROJECT.md`，把它作为项目解释和版本判断的重要来源之一。

### 版本判断时

如果某个功能不属于当前版本目标，PCP 应更自然地把它放进：

- `Backlog`
- 或 `Concern Log`

而不是继续把所有东西都塞进当前主线。

## 这轮明确不做

- 不做自动生成/自动刷新
- 不做 JSON 双文件
- 不做复杂差异比对
- 不做“自动锁定文件”机制

## 下一步建议

下一轮再实现最小版本：

1. 增加 `PROJECT.md` 模板
2. 让 intake 能读取它
3. 把版本基线嵌进 `PROJECT.md`
4. 普通执行流默认只读，不自动改写
