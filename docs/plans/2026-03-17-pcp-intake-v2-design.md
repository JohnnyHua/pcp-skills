# PCP Intake v2 设计

## 目标

把 `pcp_intake` 从“读取 handoff sidecar”扩展成统一的项目接手入口，但不自动开工。  
它的职责是：

1. 接手项目
2. 建立基础理解
3. 对齐当前 PCP 流程状态
4. 停在“等待用户决定下一步”

## 核心原则

- `intake` 不等于 `plan`
- `intake` 不等于自动执行
- `handoff` 是最佳输入，但不是唯一输入
- 默认只给轻摘要，不无脑展开所有记录
- 用户可以先提问、纠正、补充，再决定是否继续

## 三种接手情况

### 1. 有 handoff

- 读取 `HANDOFF.md` / `HANDOFF.json`
- 结合当前项目状态恢复上下文
- 输出接手摘要

重点展示：
- 当前主任务
- 队列与待审批项
- backlog / concern / worklog 的轻摘要
- 下一步建议

### 2. 没 handoff，但有 PCP

- 先做轻量项目扫描
- 再读取当前 PCP 状态
- 向用户展示“当前已有流程是什么”
- 由用户决定是否沿用这套流程

这里不自动继续执行。

### 3. 没 handoff，也没 PCP

- 做轻量项目扫描
- 给出项目概况
- 明确告诉用户：这个项目还没接入 PCP
- 只提示下一步建议，不自动进入 `plan`

## intake 时如何处理辅助记录

这些记录默认都只做“存在提示 + 一句话说明 + 轻摘要”：

- `Worklog`
  - 最近开发过程记录
- `Backlog`
  - 暂时不做但不能丢的事项
- `Concern Log`
  - 未来某阶段要重新讨论的问题
- `Changelog`
  - 最近已经完成的结果摘要

默认策略：
- 不整段展开
- 只告诉用户“有这些记录”
- 概括里面大概是什么
- 再问用户要不要深入查看

这样可以控制 token，用最小成本建立上下文。

## 统一输出结构

`pcp_intake v2` 不管来源如何，统一输出 4 段：

1. 接手来源
   - handoff / existing PCP / plain repo
2. 项目概况
3. 当前流程状态
4. 下一步建议

## 下一步建议的边界

`pcp_intake` 只做建议，不直接推进。

例如：
- 如果要继续现有流程，我可以下一步帮你恢复并查看状态
- 如果要重新开始，我可以下一步帮你规划第一轮任务
- 如果你想先确认项目理解，我们可以先问答澄清

## 本轮不做

- 深度代码理解
- 自动 planner
- 自动 adoption
- 自动进入执行
- 多源冲突判断

## 最小实现建议

第一版实现只需要做到：

1. `pcp_intake` 能识别三种来源
2. 能输出统一的接手摘要
3. 能对 `worklog / backlog / concern / changelog` 做轻摘要
4. 结束在“等待用户下一步决定”

## 一句话结论

`pcp_intake v2` 是项目接手与认知对齐入口，不是自动开工入口。
